STEP 1 — SPLUNK INSTALLATION
------------------------------
Splunk Enterprise 10.2.2 was downloaded from splunk.com and installed
on the Windows 11 VM using the default installation options. The
installer configured Splunk to run as a local non-administrator virtual
service account and created a Start Menu shortcut. An admin account was
created during installation. After installation Splunk automatically
launched and opened at localhost:8000 in the browser.

Splunk runs as a web application accessible via browser on port 8000
by default. This means no additional client software is needed —
analysts interact with the SIEM entirely through the browser interface,
which is how most enterprise Splunk deployments work.

STEP 2 — WINDOWS EVENT LOG INGESTION
--------------------------------------
Three Windows Event Log channels were configured as data inputs:
- Application
- Security
- System

These were added via Settings > Add Data > Monitor > Local Event Logs.
Splunk immediately began ingesting historical and real-time events from
all three channels. Within minutes 5,514 events were available and
searchable in Splunk covering March through April 2026.

The Security channel is the most important for SOC work — it contains
authentication events, privilege escalation, account management, and
policy changes. The System channel captures service starts/stops and
hardware events. The Application channel captures software-level events.

STEP 3 — FIRST SPL QUERY: SUCCESSFUL LOGONS
---------------------------------------------
SPL Query: source="WinEventLog:Security" EventCode=4624

Result: 524 successful logon events returned.

EventCode 4624 is Microsoft's standard successful logon event. Every
time a user authenticates to Windows this event is generated in the
Security log. 524 events across the time range confirms the Security
log is flowing correctly into Splunk and the data is searchable.

Key fields visible in results:
- Account_Name — which account logged in
- ComputerName — WIN-9FV73BAMF00
- Authentication_Package — authentication method used
- EventType=0 — success

In a real SOC investigation searching for 4624 events helps establish
normal authentication baselines — who logs in, when, and from where.
Deviations from baseline (logins at unusual hours, from unusual
locations, using unusual authentication methods) are red flags.

STEP 4 — FAILED LOGON DETECTION
---------------------------------
SPL Query: source="WinEventLog:Security" EventCode=4625

Three failed logon attempts were deliberately generated by entering
incorrect passwords at the Windows lock screen. Splunk detected all
three events immediately and returned them in the search results.

EventCode 4625 is Microsoft's failed logon event. It is one of the
most important event IDs for SOC analysts because a spike in 4625
events is the primary indicator of a brute force attack against
user accounts.

INTERESTING FINDING — BRUTE FORCE DETECTION BASELINE
The initial search returned 0 results — confirming no unauthorized
access attempts had occurred on this clean machine. After deliberately
generating 3 failed logons the events appeared immediately in Splunk,
demonstrating real-time detection capability. In a production SOC
environment an alert rule would fire automatically when 4625 events
exceed a threshold (e.g. 10 failures in 60 seconds) triggering an
immediate investigation.

STEP 5 — SPL STATISTICS QUERY
--------------------------------
SPL Query: source="WinEventLog:Security" EventCode=4625
           | stats count by Account_Name

Result:
Account_Name     Count
Bando            3
WIN-9FV73BAMF00$ 3

The stats command aggregated all failed logon events by account name
showing exactly which accounts had failed authentication attempts and
how many times. This is the standard triage query a SOC analyst runs
when investigating a potential brute force alert — it instantly
identifies the targeted account and the volume of attempts.

The pipe character (|) is fundamental to SPL — it chains commands
together passing the output of one search into the next command for
further processing. This is how analysts build increasingly refined
queries to extract actionable intelligence from raw log data.

STEP 6 — SECURITY DASHBOARD BUILT
------------------------------------
A live security dashboard named "Windows Security Monitor" was created
in Splunk with a Failed Logon Attempts panel showing account names and
failure counts over the last 24 hours.

The dashboard updates automatically as new events are ingested —
meaning any new failed logon attempt would appear on the dashboard
in real time without running a manual search. This is how SOC analysts
monitor for threats continuously rather than reactively.

Dashboard details:
Name: Windows Security Monitor
Panel: Failed Logon Attempts
Query: EventCode=4625 | stats count by Account_Name

WHAT I LEARNED
---------------
This lab demonstrated the core SIEM workflow that SOC analysts perform
daily. Installing and configuring Splunk, connecting it to live Windows
event logs, writing SPL queries to find specific event types, and
building dashboards for continuous monitoring are all skills that
directly map to tier 1 and tier 2 SOC analyst job responsibilities.

The most important lesson was the detection pipeline — an event occurs
on the endpoint (failed logon), Windows generates a log entry, Splunk
ingests it in real time, an SPL query surfaces it, and a dashboard
visualizes it. Understanding each step in this pipeline is what
separates analysts who can build detection capabilities from those
who can only use pre-built ones.
