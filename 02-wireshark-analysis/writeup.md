STEP 1 — INTERFACE SELECTION AND CAPTURE START
-----------------------------------------------
Wireshark was launched using sudo wireshark from the terminal. The eth0 
interface was selected as it is the active network interface assigned 
IP address 10.0.2.15 by VirtualBox NAT. Wireshark confirmed capture 
started and began logging all packets on the interface in real time.

Launching Wireshark with sudo is required on Linux to allow raw packet 
capture. Without elevated privileges the interface list appears but 
capture cannot start — a common first-time issue for analysts new to 
Linux environments.

STEP 2 — LIVE TRAFFIC GENERATION
----------------------------------
A ping command was run from the terminal to generate initial traffic:

    ping -c 4 google.com

This produced both DNS resolution traffic and ICMP echo request/reply 
pairs — two distinct protocol types visible immediately in the live 
capture. The packet list populated with colored rows showing the mixed 
traffic types Wireshark captured in real time.

INTERESTING FINDING — TTL OS FINGERPRINTING
The TTL value of 64 is a Linux/Unix signature. Different operating 
systems use different default TTL starting values:
- Linux/Unix = 64
- Windows = 128  
- Cisco/Network devices = 255

This means a SOC analyst can identify the likely operating system of 
a machine just from its TTL value in a packet capture — without running 
a dedicated OS detection scan. A workstation showing TTL 128 on outbound 
traffic is almost certainly Windows. A server showing TTL 64 is likely 
Linux. Any deviation from expected TTL for a known asset is worth 
investigating.

STEP 5 — DNS TRAFFIC ANALYSIS

Display filter applied: dns

Pings were run against github.com and tryhackme.com to generate DNS 
resolution traffic. 18 DNS packets were displayed showing the complete 
resolution process for all three domains.

Key observations:

google.com resolution:
- A record query (IPv4) and AAAA record query (IPv6) sent simultaneously
- Response returned 64.233.185.113

github.com resolution:
- Response returned 140.82.114.4
- SOA record also returned with authoritative nameserver dns1.p08.nsone.net

tryhackme.com resolution:
- Response returned 172.66.164.239 and 104.20.29.66 (load balanced)
- IPv6 AAAA response also returned

INTERESTING FINDING — PTR REVERSE LOOKUPS
PTR (pointer) records were visible in the capture — packets showing 
reverse DNS lookups where the machine queries for the hostname associated 
with an IP address it just resolved. This is normal automatic behavior 
but in a SOC context unexpected PTR queries from a workstation can 
indicate reconnaissance activity — a machine trying to map IP addresses 
back to hostnames without a user explicitly requesting it.

DNS runs over UDP protocol (Protocol 17) on port 53 — confirmed in the 
packet details pane.

STEP 6 — NMAP PORT SCAN CAPTURED IN WIRESHARK
-----------------------------------------------
An Nmap SYN scan was run against the gateway while Wireshark was 
actively capturing:

    sudo nmap -sS 10.0.2.2

Display filter applied: tcp.flags.syn == 1 && tcp.flags.ack == 0

Result: 1000 SYN packets displayed — exactly matching Nmap's default 
1000 port scan.

This is the visual signature of a SYN stealth scan from the defender's 
perspective. The pattern is unmistakable:
- Every packet originated from 10.0.2.15 (Kali)
- Every packet destined for 10.0.2.2 (gateway)
- All packets identical size — 58 bytes
- Destination ports rapidly cycling across hundreds of port numbers
- All happening within milliseconds

INTERESTING FINDING — NMAP SYN SCAN FINGERPRINTS
Two specific Nmap fingerprints are visible in this capture that IDS 
systems use to detect Nmap specifically:

1. TTL: 40 — Nmap deliberately uses a lower TTL than normal Linux 
   traffic (64) on SYN scan packets. Seeing TTL 40 on a flood of 
   SYN packets is a strong Nmap indicator.

2. Win=1024 — Nmap uses a non-standard TCP window size of 1024 on 
   SYN packets. Real browser and application connections use much 
   larger window sizes (typically 64240 or higher). Any IDS rule 
   looking for Win=1024 combined with SYN flags will flag Nmap 
   immediately.

In a corporate SOC environment this pattern would trigger alerts in 
Snort, Suricata, or any properly configured IDS within seconds of 
the scan starting.

STEP 7 — PCAP FILE SAVED
--------------------------
Capture saved to: ~/Desktop/wireshark_nmap_lab.pcapng

The pcapng format is the standard for Wireshark capture files and is 
universally accepted by network forensics tools. Saving captures as 
evidence artifacts is standard practice in incident response — a PCAP 
file is a court-admissible evidence artifact in many jurisdictions and 
is the primary format for sharing network evidence between analysts.

STEP 8 — TCP STREAM ANALYSIS
------------------------------
Analyze menu — Follow TCP Stream applied to stream 9.

Result: Two packets displayed showing the complete conversation for 
one Nmap probe on port 587:

Packet 57: 10.0.2.15 → 10.0.2.2 [SYN] — Nmap knocking on port 587
Packet 58: 10.0.2.2 → 10.0.2.15 [RST, ACK] — Gateway rejecting it

This is the complete lifecycle of one port probe — request and 
rejection in two packets. Following TCP streams is one of the most 
powerful Wireshark features for SOC analysts because it reconstructs 
entire conversations including HTTP requests, login attempts, and 
data transfers in readable format.

STEP 9 — RST PACKET ANALYSIS
------------------------------
Display filter applied: tcp.flags.reset == 1

Result: 1000 RST ACK packets displayed — all highlighted red by 
Wireshark's default color coding.

Every packet in this view is the gateway responding to an Nmap SYN 
probe with a reset — meaning the probed port is closed and no service 
is running there. The wall of red RST packets is the visual confirmation 
that the gateway rejected all 1000 port probes.

Combined with the SYN flood captured in Step 6, this gives the complete 
picture of the port scan from both directions - 1000 SYN packets going 
out and 1000 RST responses coming back.

INTERESTING FINDING - RED COLOR CODING AS SOC ALERT INDICATOR
Wireshark's automatic red color coding of RST packets mirrors how SIEM 
dashboards highlight suspicious traffic. A wall of red in any network 
monitoring tool is an immediate visual signal requiring investigation. 
Training your eye to recognize these color patterns in Wireshark directly 
translates to faster alert triage in production SOC environments.

WHAT I LEARNED
---------------
This lab demonstrated that Wireshark is not just a troubleshooting tool 
but a critical SOC investigation platform. The ability to capture traffic 
while simultaneously running an attack tool (Nmap) and then analyze the 
results from the defender's perspective is a core dual-skill that most 
entry-level candidates have not practiced.

Key takeaways:
- TTL values can fingerprint operating systems without running dedicated 
  scans — useful for quick asset identification during investigations
- Nmap leaves specific fingerprints (TTL 40, Win=1024) that IDS systems 
  actively look for — understanding attacker tool signatures makes you 
  a better defender
- PTR reverse lookups can indicate automated reconnaissance
- RST packets in bulk are a reliable port scan indicator
- PCAP files are professional evidence artifacts used in real incident 
  response and legal proceedings
- Display filters are non-destructive — always filter rather than delete
